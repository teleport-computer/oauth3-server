oauth3

the delegation plugin · read access to your accounts without surrendering your cookies · with tinycloud · 2026

sealed rooms

Teleport builds with TEEs — trusted execution environments, sealed rooms that even the machine’s operator can’t see into. One bet runs through every teleport experiment: once anyone can spin up a room no one can reach into, the things we trust to passwords, platforms, and OAuth scopes can move somewhere you actually control. Teleport.Best put one-shot identity in a room. Login With Anything put authentication in one. oauth3 puts your credentials there.

the problem

Most of the web has no OAuth. To let an app read your Otter transcripts or your YouTube history, you hand over the whole cookie jar — and with it, everything that jar can do. “Log in with X” was supposed to scope that down; in practice it hands over a standing session, and the rest of the web just takes your raw cookies. Delegation collapses into surrender.

what oauth3 does

oauth3 breaks the handoff into three pieces, with the jar sealed inside a room you control:

  • an extension syncs your whole cookie jar for a site into the sealed room;
  • a plugin turns that jar into clean list / fetch reads for one site’s unofficial API;
  • a scoped token lets an app or agent make those reads — carrying the holder’s identity, never the jar.

Revoke the token and the access is gone. The jar never left the room, and the app never held it. Running on TinyCloud’s signed-invocation substrate, the room is multi-tenant — a sovereign auth layer where the user’s keys, not the platform’s, decide who gets what.

how it works

In production the server is a tee-daemon project on a dstack node; the jar is sealed at rest (AES-GCM) and a background loop polls each plugin on a schedule.

two paths

Most sites expose an unofficial API a plugin can read — cheap, no browser, and any federated oauth3 instance can serve it. (Federation is coming soon.)

Some sites have no usable API at all. There the sealed room drives a real browser — generic login-with-anything screenshots, or login by TikTok QR code with no plugin written at all. That path is expensive, so it’s metered: you spend credit to use it.

use it

  1. get your jar in

    Two ways — you don’t need the extension. With it: load oauth3-extension unpacked, set the instance URL + owner secret, pick a plugin, Sync jar now (then it auto-syncs on cookie change and every 30 min). Without it: copy the cookie from devtools and POST it yourself — one curl, no install.

    # no extension — paste the cookie(s) for the plugin's site
curl -s $NODE/api/cookies -H "Authorization: Bearer $OWNER_SECRET" \
  -H 'Content-Type: application/json' \
  -d '{"plugin":"otter","cookies":{"sessionid":"…","csrftoken":"…"}}'

  2. mint a scoped read token

    With your OWNER_SECRET, mint a token for one plugin and subject. The token is all the app needs — it never sees the jar.

    # mint a scoped read token for the otter plugin
curl -s $NODE/api/tokens -H "Authorization: Bearer $OWNER_SECRET" \
  -H 'Content-Type: application/json' -d '{"plugin":"otter","subject":"andrew"}'

  3. read with the token

    The app or agent calls the plugin’s read API with the scoped token. No raw cookies, ever.

    # list, then fetch one item
curl -s $NODE/api/otter/items        -H "Authorization: Bearer tok-otter-…"
curl -s $NODE/api/otter/items/<otid> -H "Authorization: Bearer tok-otter-…"

ways to use it

A handful of distinct paths, each held to its own UX bar — tracked in USER-JOURNEYS.md.

  • Paste a cookie, no install — one CLI flow: verify → sync → token → read.
  • Install the extension — sync once, then forget about cookies; it auto-refreshes.
  • Grant an app — approve a listed app onto your jar, read with a scoped token, revoke anytime.
  • Capture a page — for JS-gated sites, render with your jar and grab a screenshot / DOM.
  • Add a site — copy the adapter template, fill the endpoints, register.
