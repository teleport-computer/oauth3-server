what it is

You build an app that reads people’s accounts — watch history, meeting notes, whatever — push it to your pod and share the link, same as Vercel. The difference: it runs in an attested enclave. Where this is headed: anyone who opens the link can verify the running code is exactly what you published, and that it touches only the accounts and scopes you declared. You ask for access in plain words — “reads your watch history, nothing else” — and the deployment backs the claim. People say yes because your app is legible, not because they trust you. And you never hold their credential: it stays sealed in their instance, and your app reads through a scoped token. where the proof story is going →

For the people who use your app, the same machinery reads the other way: a handful of sites offer OAuth, but everywhere else, letting an app use your account is all-or-nothing — hand over your password, or your whole cookie jar. oauth3 turns any account you can log into into one you can delegate: pick what an app gets, a single read or the run of the account, and change your mind whenever.

how it works

  • your cookie jar for a site goes into the instance (via the extension, or one paste) — sealed, never handed out;
  • an app asks to read that site → you approve once → it gets a scoped token;
  • the app reads through the instance with that token. Revoke it and the app is done. The jar never moved.

use it

with the extension

one click

Install oauth3-extension. On a supported site, open it and hit Save — the jar syncs and stays fresh on its own.

no extension — in your terminal

paste a cookie

Copy the cookie from DevTools and run the CLI — no install, no browser automation:

# in a terminal
oauth3 sync otter --cookie 'sessionid=…'
oauth3 token otter            # → scoped token
oauth3 read  otter            # real data, no cookie

for developers

Build an app with the SDK: connect() for the user’s approval, then list() / fetch() with the token. New sites are an adapter template in the server. The raw API lives at the instance’s /oauth3/api.

why trust it

The instance runs inside a TEE — a sealed room the operator can’t see into. The host is the tee-daemon, which attests the exact code each app runs; an app (or the extension) can pin that measurement before trusting an instance — you trust the code, not the operator. A teleport experiment, with TinyCloud.
